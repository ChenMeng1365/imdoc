
# 这是一个可执行Markdown扩展文档的例子

文档的描述随便写，不影响markdown原本的语意。
以下是给Markdown一些增强的功能：

## 1. 全局配置

如果要做一些全局配置参数，使用如下shell，并以“.yml”、“.yaml”结尾（其实也可以没有后缀，但内容格式一定要是YAML全局散列）。

```shell
#conf>配置文件名.yml
```

它的内容应该形如：

```shell
--- # 文档名称
配置名1: 属性值1
配置名2: 属性值2
配置名3: 属性值3
………………
```

全局配置的好处就是渲染表格时，它不会渲染出来，可以隐藏一些信息。 如果需要检查参数设置，可以以打印代码的方式查阅，默认占用表名"global"，参考`EnData.table('global')`。

没有全局配置也没有任何问题，找不到配置文件全局变量会置空。

## 2. 内置表格

如下编辑会生成一个表格：

((表名))

|字段名1 |字段名2 |字段名3  |…………|
|--------|-------|---------|----|
|字段值1 |字段值2 |${绑定名}|…………|

如果没有表名，那么会生成一个Anonymous的表。

使用`${绑定名}`可以生成一个引用名和引用内容的绑定，用以替换复杂的文本内容，生成表格时会自动绑定替换。

## 3. 外部表格

使用如下shell导入一个外部表格文件：

```shell
#output>表名.csv
#row>1
#col>path,server,protocol,access-key
```

内置表名会是"表名.csv"。
由于文件格式支持问题仅支持CSV文件引入，其他格式要自行编写代码读取和解析。

可以追加`#row>`和`#col>`用来筛选、投影特定的行和列，构成新的子表。
行筛选通过行号，可以是数字，也可以是范围，比如`1,2,4,11~17`。
列投影投影的是表头字段名，比如`field1, field2 ... field65535`。

## 4. 代码输入&输出通道

使用如下shell来输入一段代码：

```shell
#input>代码段名
#src>解释引擎

书写你自己的代码
………………
```

使用如下shell来展示代码执行的结果：

```shell
#output>代码段名
写什么无所谓，渲染后这一段会被替换掉
```

input管道下的代码，执行后会被输出到output管道下。

代码根据“代码段名”关联，可以目录/树状命名，但不推荐以相同前缀子串整体做命名（比如一个是“/path/to/folder”另一个是“/path/to/folder/subfolder”，暂未修复<b>[TODO]</b>）。

代码解释引擎就是常见的语言或命令行工具。
默认支持lm、tm、ruby、python、erlang和bash，其他语言可以基于shell扩展，但windows平台要注意cmd、ps本身和bash有差异，执行并不稳定。
多版本/环境管理解释器的执行主要取决于渲染文档时的shell环境，比如执行python使用conda构建环境，那么调用`cm`命令时shell需要能进入并选择env（改造启动脚本或sublime构建工具）。

## 5. 数据绑定

表格的数据是默认和代码数据段绑定的，这个特性基于ruby的`endata`开发，也可以应用到lm、tm等工具上。

配置文件会以全局变量的形式记录在`$_tmp_global_`中。
表格一行是一条记录，以全局变量的形式记录在`$_tmp_endata_`。

在ruby中，数据直接放置在`__END__`中，使用`EnData/ApiRequest`的`EnData::table`来导入表，全局变量则放在`global`这样一个特殊的表中，注意EnData的数据定义是有顺序的（见`EnData/api-app.rb`）。

在erlang中，数据以函数的形式附着在escript末尾，我们将其做成prolog等效的事实函数（形如`fact(A,B,C,...)->{D,E,F}.`）。
